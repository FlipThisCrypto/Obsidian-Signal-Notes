MonkeyZoo Production Workflow: The Expressions Gallery Works on GitHub Pages, and Issue 01 Has a Pipeline

Originally shared by on X · retrieved 2026-07-16. This is added context and credit, not the original work.

I have been building out MonkeyZoo Production Workflow System, a repo where I am figuring out, in public, how to actually run production on a MonkeyZoo comic book series. Not just draw the pages, but build the system underneath them: character references, canon tracking, issue pipelines, all of it. The last 24 hours had two solid pieces of progress.

What just shipped

Expressions gallery now works on GitHub Pages. The Expressions gallery is where character expression references live, and it was not displaying right once deployed to Pages (things that work fine locally do not always survive the jump to a live static site). That is fixed now, so the gallery actually renders the way it is supposed to when people pull up the live site instead of running it locally.

Project Map, canon media, and the Issue 01 Signal pipeline. This is the bigger one. I added a Project Map, which gives the repo an actual overview of how the pieces connect instead of making people dig through folders to understand the workflow. Alongside that, canon media went in (reference assets that need to stay consistent across the series), and I built out the production pipeline for Issue 01, the Signal season. That means the steps for getting that specific issue from concept to finished pages now have a real, trackable structure instead of living in my head.

Expressions gallery, project map, and the Issue 01 pipeline, working as one system.

Why it matters

A comic book series lives or dies on consistency: the same character looking like themselves issue to issue, the same tone, the same canon. Doing that from memory does not scale past one or two issues. Having a Project Map and canon media in the repo means the reference material is centralized and versioned instead of scattered across chat threads and local folders. And having an actual pipeline for Issue 01 means I can point at a stage and know what is done and what is next, which matters a lot more once there is an Issue 02 to worry about too.

Rick's take

The Expressions gallery fix is small but it is the kind of thing that makes a repo feel real instead of theoretical. If the live site is broken, nobody trusts the rest of it. Getting that sorted first, before piling on more features, felt like the right order of operations.

The part I am actually excited about is the Issue 01 Signal pipeline. This is the first time the workflow has a concrete issue running through it end to end, which means I get to find out what parts of my process are solid and what parts I made up and need to fix. That is the whole point of building this in public: I would rather learn that on Issue 01 than three issues deep.

What's next

With the Project Map and canon media in place, the next logical step is running more of Issue 01 through the pipeline and seeing where it holds up or breaks. I will keep logging that here as it happens.
