The Sinking Ship: 40 Tests in One Day

Originally shared by on X · retrieved 2026-07-23. This is added context and credit, not the original work.

I've been building The Sinking Ship for a while now. It's a Chia blockchain collectibles project with generative ship art, rarity mechanics, an on-chain fulfillment engine, and a full website. The pieces have been coming together. This week I did something less flashy but more important: I wrote tests. A lot of them.

What just shipped

Forty test iterations in roughly 24 hours, covering the entire stack by theme.

Infrastructure and reliability. Tests landed for the circuit breaker (snapshot and half-open probe behavior), the budget guard (zero-budget edge cases), the reconcile lock (context manager cleanup on exit), and the timing utility (slow-section warnings). These are the guardrails that keep the engine from going sideways quietly.

Ship generation. Coverage for shipgen package initialization, version checks, rarity constants, Amano Ink art primitives, config loading, schema validation, and canonical JSON serialization. Eight JSON schemas now get validated as non-empty and structurally correct on every run.

Fulfillment and chain mechanics. Tests for dry-run offer builder determinism (same inputs, same outputs, every time), the fulfillment type system, and a 404 path on the mock coinset so the fake blockchain client fails correctly on unknown paths. The DRBG (deterministic random number generator) has salt-differentiation tests now, which matters when rarity rolls need to be reproducible and auditable.

Observability. Prometheus metric export has special character escaping tests. The JSON line formatter has exception logging coverage. Health check degradation when a circuit is open is tested explicitly.

CLI scripts and web. Tests across the render engine, validate-configs, Monte Carlo simulation, site link checker, build-site-data (with a new --out option), tune-weights (also with --out), the soak script, style scorer, placeholder sprite generator, brand and share card builder, polished asset installer, art pipeline utilities, documentation link checker, and an end-to-end pipeline smoke test. A Node.js verifier for the roll vector outputs is also in.

The site SEO and security tests now validate sitemap, robots.txt, and security.txt standards. A chain identity test now rejects placeholder values outright, so stub data cannot reach production.

The loop: build a piece, test it, trust it.

Why it matters

When fulfillment touches real on-chain assets (Chia offers, actual NFTs), test coverage is not optional. A bug in the rarity roll or offer builder could mean minting the wrong thing or breaking a buyer's experience. Forty tests mean forty fewer ways the system can surprise me in production.

Rick's take

This was a heads-down sprint. I knew the code was outrunning the tests and I did not want to ship a fulfillment system without confidence in the pieces underneath it. Writing tests is not my favorite part, but debugging live problems is worse, so here we are.

What I actually found useful: several of these edge cases (zero-budget guard, 404 on the mock coinset, placeholder rejection at the chain layer) surfaced design assumptions I had not spelled out clearly. That is worth something on its own, separate from the coverage number.

What's next

With coverage solid, the next move is running the fulfillment engine against real testnet conditions. The Admiral chest tier has its quantity ceiling locked in at 300. The site data pipeline now has configurable output, which means the publish step is getting close.
