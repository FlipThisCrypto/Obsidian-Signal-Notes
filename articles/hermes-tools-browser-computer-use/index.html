The Hermes Tools That Work Where APIs Stop

What was bookmarked

Tony Simons (@tonysimons_) published Part 9 of his Hermes Agent Master Class as a full X Article. The piece covers two new capability categories: browser automation and computer use. Browser automation turns Hermes into an actual Chromium browser that can navigate real pages, click real elements, fill real forms, and read each page as a structured accessibility tree. Computer use extends that further, letting the agent interact with any visible window on macOS, Windows, or Linux without moving your physical cursor or shifting your focus.

Same agent, two different surfaces. Browser runs isolated in Chromium. Computer use runs on your actual desktop.

Why it matters

It is relatively rare to see an agent series work through the full capability stack this carefully, and browser use and computer use are the layers where things get genuinely interesting. Browser automation and computer use address a real constraint: agents that only speak to APIs are blocked the moment the interface they need has no API to call.

The distinction Tony draws between the two tools matters in practice. Browser automation runs inside an isolated Chromium instance, separate from your machine. The agent gets its own session with its own cookies, cache, and fingerprint. It cannot touch the rest of your system, and the session cleans up after inactivity. That isolation makes it the right tool for web tasks: sites that depend on JavaScript rendering, forms that require a real browser session, content that blocks API scrapers.

Computer use operates on your actual desktop. That is a larger surface area, which is why it comes with more guardrails. But it also means the agent can reach things with no web equivalent: native applications, local dialogs, system settings. If it is visible on screen, Hermes can probably interact with it.

One detail worth calling out is the hybrid routing feature. With a cloud provider configured, public URLs go through the cloud browser while localhost and private network addresses route automatically to a local Chromium instance. The agent can screenshot your local dev server and scrape a live website in the same session without switching providers manually. For anyone testing builds against live data, that matters.

The practical breakdown

What the browser toolset does: The agent reads each page as an accessibility tree, a text-based snapshot of every interactive element with numbered reference IDs. It clicks by ID rather than guessing at pixel coordinates, which is meaningfully more reliable. Five backends cover different needs: Browserbase provides managed cloud browsers with residential proxies and CAPTCHA handling; Browser Use and Firecrawl are alternative cloud providers with their own anti-detection features; local Chromium via the DevTools Protocol lets the agent attach to your existing Chrome or Brave install; and local mode uses a Chromium install driven by the agent-browser CLI.

What computer use does: The agent captures any visible window and overlays numbered elements on every interactive item. Your physical cursor does not move. The target window does not come to front. Virtual desktops do not switch. Each session gets its own cursor identity, so concurrent agents stay clean and do not produce conflicting actions. The article notes that computer use works with any tool-capable model, including local endpoints, since the toolset communicates over MCP to cua-driver, an open-source background driver that handles the platform-specific accessibility stack.

Who it helps: Builders running workflows that touch interfaces without APIs. Developers testing against local builds. Anyone who has hit a wall because the tool they needed to automate only ships as a native app or a JavaScript-heavy web page.

What still needs work: Screenshots are expensive. The article puts a single screenshot at approximately 1,500 tokens. Hermes applies four optimizations to keep this manageable: keeping only the three most recent screenshots in context, stripping old image data from tool results with a context compressor, counting images at a flat 1,500-token rate regardless of base64 length, and clearing old tool results server-side on Anthropic. A full session lands around 30,000 screenshot tokens rather than what could be 600,000 without those controls. That gap is what makes desktop agents practical instead of theoretically interesting.

What I would test first: The web form signup example from the article is the clearest entry point. The agent navigates to a signup page, reads the accessibility tree, identifies the email and password fields, fills them, clicks submit, and takes a screenshot to confirm success. That is a concrete, bounded task that demonstrates the capability without needing a complex setup.

Rick's take

What I keep coming back to in this series is that Tony is working through the stack in order, and this installment is where things get genuinely interesting. Text-only agents are useful, but they run out of road fast once the task involves a real interface. Browser automation solves the web half of that problem. Computer use solves the rest.

The background execution model is the part that changes how you think about this. Your cursor stays put. Your focus does not shift. The agent works alongside you without interrupting your session. That is a different relationship than most people picture when they imagine AI automation, and I think it is the right one.

The token efficiency work is also worth appreciating. Getting from a theoretical 600,000 tokens down to 30,000 per session is not a minor optimization. It is the engineering that determines whether this is a practical daily tool or something that only makes sense in demos. Looking forward to Part 10.
