The Sinking Ship Hits Iteration 50: Trace IDs, Versioned Migrations, and a CI Gate That Holds

Originally shared by on X · retrieved 2026-07-24. This is added context and credit, not the original work.

I have been building The Sinking Ship in public, one iteration at a time. This week it crossed iteration 50, and I closed that milestone with a CI coherence gate that checks whether all 50 prior changes still hold together. Then round 2 started immediately: trace IDs on fulfillment events and a versioned migration framework for the SQLite ledger. Three changes, all pointing at the same thing.

[HERO IMAGE]

What just shipped

Milestone: CI coherence gate (iteration 50)

Iteration 50 landed with test_full_ci_gate.py, a test suite that validates coherence across all 50 iterations as a whole. This is not unit testing in isolation. It is a check that everything I have built still works together without silent regressions hiding in the seams. Closing a milestone by proving the whole thing is still standing felt like the right way to do it.

Incident correlation: UUID4 trace IDs on fulfillment events (round 2, iteration 1)

Every fulfillment tick event now carries a UUID4 trace_id. When something goes wrong in a system like this, you want to be able to grep a single ID and reconstruct exactly what happened across the log. Without it, you are reading tea leaves. With it, you have a thread to follow.

Database versioning: migration framework for SqliteLedger (round 2, iteration 2)

The SQLite ledger now has a proper versioned migration framework. The first migration under it (v2) adds an archived_at column, which enables soft-delete and archival on ledger rows without losing history. More important than the column itself is having the framework. Future schema changes are now structured operations, not manual risks.

Three changes. One direction: software that holds up over time.

Why it matters

Trace IDs, schema migrations with versioning, and a CI gate that proves cross-iteration coherence are not glamorous features. They are the infrastructure of software that is meant to run over time, not just pass a demo. These three changes together are what building in public actually looks like when you are being honest about what the project needs.

Rick's take

I named this project The Sinking Ship partly as a joke about entropy in software, and partly as a genuine reminder that nothing ships unless you fight for it. Every iteration is a patch on the hull.

Hitting 50 iterations was a real moment for me. The CI gate I wrote to close it has to pass across all 50 prior changes. That is what made it feel like a milestone rather than just a number. It is proof that I have not been silently breaking things as I go.

The trace ID work is something I had been putting off, and I am glad round 2 started there. The second something goes sideways in a fulfillment ledger system, you need to be able to trace it. Now I can. The migration framework is the other piece I knew I needed before the ledger got any more complex. Both of these were the right first moves for round 2.

What's next

Round 2 is underway and the foundation is cleaner than round 1 ever was. The migration framework and trace infrastructure make it possible to add more complex fulfillment behavior without the data risk that came before. The direction is richer auditability and more structured tracking. More iterations coming.
